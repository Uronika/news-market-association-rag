# 项目技术拆解、局限性与可扩展性

## 一、项目定位

本项目是一个面向课程研究的“新闻舆情—行情异动关联解释型 RAG”原型。它接收自然语言问题，检索近期新闻与日线行情，识别异常波动窗口，组织企业、行业、主题和来源之间的关系，再由大语言模型生成带引用和风险边界的关联解释。

系统解决的是信息组织与解释问题，不是交易决策问题。其输出固定限定为 `association_only`，不应被理解为因果证明、股价预测或投资建议。

## 二、核心结构拆解

| 层次 | 主要模块 | 当前职责 | 成熟度判断 |
| --- | --- | --- | --- |
| 接口层 | `app/main.py`、`app/schemas.py` | 提供健康检查、策略说明、同步/异步分析、进度查询和结果查询接口 | 课程原型可用 |
| 编排层 | `app/rag.py` | 组织问题解析、数据获取、过滤重排、图谱、LLM、守卫、评价和持久化 | 功能完整，但模块过于集中 |
| 数据源层 | `app/sources.py` | 适配 Yahoo Finance、GDELT、Stooq、Alpha Vantage、Twelve Data 等来源 | 接口可扩展，稳定性依赖外部服务 |
| 分析层 | `app/analytics.py`、`app/insights.py` | 计算收益率、成交量异常、行业映射、企业关系和情景假设 | 主要依赖规则与启发式方法 |
| 图谱层 | `app/graph.py` | 构造请求级 NetworkX 图并返回可视化节点、边和注释 | 属于临时关系图，不是持久化知识图谱 |
| 生成层 | `app/llm.py` | 调用 DeepSeek，根据证据生成中文报告；失败时返回明确状态 | 单模型依赖明显 |
| 风险层 | `app/guard.py` | 限制交易建议、目标价、预测和确定性因果表述 | 有基本边界，但规则较脆弱 |
| 存储层 | `app/db.py`、`app/models.py`、`migrations/` | 保存公司、行情、新闻、分析结果和引用 | 可用于单机课程演示 |
| 展示层 | `frontend/index.html` | 展示报告、行情、时间线、词云、图谱、检索流程和评价结果 | 无构建依赖，便于演示；维护成本较高 |

一次分析请求的主要数据流为：问题识别 → 时间窗和候选量推断 → 行情与新闻检索 → 异动检测 → 规则重排 → 推理上下文与临时图谱 → LLM 生成 → 风险守卫 → 评价与引用 → 前端展示。

## 三、主要局限性

### 1. 数据覆盖和稳定性不足

真实新闻与行情依赖公开或第三方接口。Yahoo Finance、GDELT 和 Stooq 可能出现限流、反爬验证、字段变化或历史覆盖不完整，系统无法提供生产级服务等级保证。新闻目前主要保存标题、链接和短摘要，缺少全文、版权状态、来源可信度和更严格的事实核验。

### 2. 检索仍以规则和关键词为主

Embedding 默认关闭，当前重排主要依据时间距离、企业词命中、异常窗口、摘要存在性和标题去重等规则。其优点是透明、成本低，但对同义表达、跨语言语义、简称歧义和隐含关系的识别能力有限。候选新闻至少取 100 条，但进入推理、图谱和引用的数量被固定上限截断，数量增加不等于相关性提升。

### 3. 企业识别和行业映射不够系统

多企业识别依赖有限的内置企业词表和关键词规则，尚未接入权威证券主数据、统一代码体系、公司别名库和实体消歧模型。行业影响由关键词映射得到，适合展示，但不能替代行业研究中的供应链、收入暴露度和地区暴露度数据。

### 4. 图谱是解释性视图，不是事实知识库

当前图谱在每次请求中临时生成，节点和边主要来自共现、主题词和规则映射。边表示“可能有关联”，不表示已验证事实或因果关系。部分新闻与行情窗口之间的连接较宽泛，尚未对时间距离、证据强度、关系方向和有效期进行严格建模。

### 5. LLM 输出存在依赖、成本和一致性风险

生成环节主要依赖 DeepSeek。外部模型不可用时，系统只能返回检索结果或受控报告，无法保证完整生成。长上下文会增加响应时间与成本；标题和短摘要也可能不足以支持深入判断。当前没有强制结构化生成、逐句证据校验和多模型交叉验证，报告质量仍受提示词和模型版本影响。

### 6. 风险守卫和评价仍是原型级

风险守卫主要采用词语检测和替换，难以覆盖隐晦的交易建议、复杂因果暗示和上下文变体。置信度主要由引用数量、行情覆盖和候选数量推断，不等同于统计置信度。项目尚未建立人工标注基准，也没有系统计算 `Precision@K`、`Recall@K`、`NDCG`、引用一致性和事实忠实度。

### 7. 并发和可靠性能力有限

异步任务进度保存在进程内存中，服务重启后任务状态会丢失，多进程部署时也无法共享。数据库表虽然建立了查询索引，但缺少完整的唯一约束和并发幂等设计，极端情况下仍可能重复写入。系统没有消息队列、分布式缓存、熔断器、集中日志、指标监控和链路追踪。

### 8. 工程质量仍有明显技术债务

`app/rag.py` 承担过多职责，数据获取、业务编排、报告格式化、评价和持久化耦合较高。部分中文字符串存在历史编码损坏，需要统一 UTF-8 并增加编码检查。现有测试覆盖核心函数和 API，但缺少真实数据源契约测试、浏览器端到端测试、并发测试、压力测试和安全测试。

## 四、可扩展性分析

### 1. 数据源扩展

新闻源和行情源已经抽象为适配器接口，可继续接入合规的商业数据、交易所数据、公告、财报和社交媒体数据。扩展时应增加统一字段规范、来源优先级、重试与指数退避、限流、缓存有效期、数据血缘和授权信息。

### 2. 检索能力扩展

可升级为混合检索：关键词/BM25 负责精确命中，向量检索负责语义召回，元数据过滤负责时间、市场和来源约束，再由交叉编码器或 LLM reranker 重排。向量库可选 PostgreSQL `pgvector`，避免过早引入独立基础设施。

### 3. 实体与图谱扩展

可建立证券主数据表、别名表、行业分类和关系表，并为每条关系保存类型、方向、证据、置信度和有效时间。数据规模扩大后，可选择 PostgreSQL 图查询扩展或 Neo4j 等图库，但只有在跨请求复用和复杂路径查询成为真实需求时才有必要。

### 4. 评价体系扩展

应建立小规模人工标注集，分别评价实体识别、新闻相关性、时间窗匹配、关系抽取、引用正确性和回答忠实度。线上可记录数据源成功率、检索耗时、LLM 延迟、失败类型和用户反馈，使“可评价”从展示字段变成可持续优化机制。

### 5. 并发与部署扩展

可将任务状态迁移到 Redis，将长分析任务交给 Celery、RQ 或其他队列；进度通过 SSE 或 WebSocket 推送。数据库增加唯一约束、事务隔离和幂等键，API 增加认证、速率限制、请求配额和审计日志后，才能支持多用户部署。

### 6. 模型与安全扩展

LLM 层可增加统一供应商接口、失败切换、结构化 JSON 输出、提示词版本管理和调用成本记录。生产环境中密钥应放入系统环境变量或专用密钥管理服务，不能由公开前端长期持有；同时需要输入长度限制、URL 安全检查、内容过滤和最小化日志策略。

## 五、建议的演进优先级

1. **先修工程基础**：统一 UTF-8、拆分 `rag.py`、补充结构化日志和端到端测试。
2. **再提升检索质量**：建立实体主数据，加入 BM25、向量检索和可评价的重排指标。
3. **随后增强可靠性**：增加缓存、重试、幂等约束、持久化任务队列和可观测性。
4. **最后扩展图谱和多用户能力**：在明确查询需求和数据规模后再引入图库、权限系统和分布式部署。

## 六、结论

本项目已经形成完整的 RAG 课程原型：具备真实数据检索、上下文增强、生成、引用、图谱展示、评价和风险约束链路。其优势是过程透明、边界清楚、演示完整；其不足是检索和图谱仍偏规则化，数据源与单一 LLM 依赖较强，评价、并发和工程治理尚未达到生产标准。

因此，合理定位应是“可运行、可解释、可扩展的教学型原型”，而不是成熟投研平台。后续扩展应优先提高数据质量、检索可测性和系统可靠性，而不是单纯增加页面元素或模型生成长度。
